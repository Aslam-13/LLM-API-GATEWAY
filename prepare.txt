  A middleman API that sits between your app and LLM providers (OpenAI, Gemini).

  your app ──▶ [our gateway] ──▶ OpenAI
                             └─▶ Gemini

  Instead of calling OpenAI directly, you call POST /v1/chat/completions on our gateway with the same OpenAI-shaped
  body. The gateway then does things OpenAI alone can't:

  Why — what value does it add?

  1. Caching — if two users ask "What's the capital of France?", we only call OpenAI once. Second request returns from
  Redis in ~1ms instead of ~1000ms. Saves money + latency.
  2. Semantic caching — even if the wording differs ("Tell me France's capital"), we embed the prompt as a vector and
  match against past prompts. Same answer, no API call.
  3. Fallback — if OpenAI is down / rate-limited, we retry, then automatically try Gemini. Your app never sees the
  outage.
  4. Auth + quotas — you issue your own API keys (sk-gw-live-…) to your users, rate-limit them, revoke them, see
  per-user usage. OpenAI's key is hidden behind ours.
  5. Observability — every request logged
  


  
  1. Storing customer data — privacy

  Yes, it's a real concern. You have three realistic models:

  ┌─────────────────┬──────────────────────────────┬───────────────────────────────────────────────────────────────┐
  │      Model      │         Who you are          │                            Storage                            │
  ├─────────────────┼──────────────────────────────┼───────────────────────────────────────────────────────────────┤
  │                 │ You ship this gateway as a   │ Data never leaves their machines — no privacy issue, they own │
  │ Self-hosted     │ Docker image, customers run  │  the Postgres                                                 │
  │                 │ it on their infra            │                                                               │
  ├─────────────────┼──────────────────────────────┼───────────────────────────────────────────────────────────────┤
  │ Managed         │ You host it, customers send  │ You must publish a privacy policy, encrypt at rest, offer a   │
  │ multi-tenant    │ their prompts to your        │ "cache opt-out" toggle per request / per key, add a retention │
  │ SaaS            │ servers                      │  policy (auto-delete after N days), possibly GDPR DPA         │
  ├─────────────────┼──────────────────────────────┼───────────────────────────────────────────────────────────────┤
  │ Internal tool   │ Your own company uses it     │ Just an ops concern, not a customer one                       │
  │ only            │                              │                                                               │
  └─────────────────┴──────────────────────────────┴───────────────────────────────────────────────────────────────┘

  Most production gateways offer a per-request cache: false header so sensitive prompts skip storage, plus a "no logs /
  PII redaction mode" for paying enterprise tiers. That's the table-stakes answer to "how can you serve customers?".
  Cache prompt-text and response are the sensitive bits — tokens, provider, latency are fine to keep.

  Also: caching is opt-in value. A customer who sees "90% cache hit rate saved us $800/month" is happy to trade some
  storage for that. The privacy story is: encrypted, scoped per API key, deletable on request, never shared across
  tenants.

  2. Why would anyone use yours?

  You're right that each SDK has some of this. The pitch for a gateway is aggregation:

  - Multi-provider with one SDK — customer writes to OpenAI's API shape and you fan out to OpenAI / Gemini / Claude /
  Azure / Bedrock with zero code changes when they want to switch. The lock-in escape is the whole point.
  - Cross-provider caching — Gemini's caching works only within Gemini. Yours works regardless of which provider you end
   up hitting.
  - One unified bill and quota — instead of 4 provider dashboards, one dashboard shows spend/errors/latency across all.
  - Team-level API keys + per-user limits — OpenAI's keys are coarse. You give Finance-team-key vs Marketing-team-key
  with separate budgets.
  - Fallback on outages — OpenAI goes down Tuesday, customer's product keeps working because you cut over to Gemini
  silently.
  - Observability — most companies end up writing this exact logging + dashboard themselves. You sell it as finished.

  Real commercial examples: Portkey, Helicone, OpenRouter, LiteLLM, Kong AI Gateway, Cloudflare AI Gateway. All real
  businesses doing exactly this. The demand is proven.

  Could a big team build it? Yes. Will they? Usually not — it's not their core product. That's the whole
  managed-infrastructure playbook (Datadog, Segment, Stripe, etc.).

  BYO is usually the default because:
  - Lower liability for you
  - No need to prepay providers
  - Customers' existing credits / enterprise OpenAI contracts still work
  - Compliance is on them

  Managed is attractive when:
  - Customer is tiny / doesn't want to deal with signups
  - You can negotiate a volume discount and resell cheaper than their retail

  For this v1 project, the simplest story is BYO: they paste their OpenAI/Gemini key into your dashboard, you store it
  encrypted (ideally in a secrets manager, not Postgres), and the ApiKey row they use with you is the one the gateway

  You'd need a small addition to the schema for that: a provider_credentials table keyed to api_key_id,
  encrypted-at-rest, containing {provider, key_plaintext_encrypted, key_hint}. Not in our current v1, but an obvious
  Phase 19 feature for the commercial version.

  What RabbitMQ did in your test

  When you hit POST /v1/embeddings/batch, here's the full flow:

  1. FastAPI   → inserts Job row in Postgres (status=pending)
                 returns {job_id, status: pending} immediately (202)
  2. FastAPI   → calls batch_embeddings_task.delay(job_id, "gemini", model, texts)
                 Celery serializes args → JSON message
                 Celery publishes message to RabbitMQ queue "celery"
  3. RabbitMQ  → holds message until a consumer (worker) pulls it
  4. Worker    → pulls message, deserializes, runs Python function
                 updates Job row (running → succeeded, stores vectors)
                 ACKs message → RabbitMQ removes it
  5. Worker    → stores return value in Redis (DB 1, result backend)


    Celery is not the queue. Celery is a Python framework that:

  1. Client side (inside FastAPI): wraps batch_embeddings_task.delay(...) so it looks like a normal function call, but
  actually serializes args → JSON → publishes to RabbitMQ. Returns immediately.
  2. Worker side (terminal B): long-running Python process that subscribes to the queue, receives messages, deserializes
   args, invokes the actual batch_embeddings_task Python function, handles retries, stores the return value in Redis,
  ACKs the message.
  3. Result backend (Redis DB 1): after the task returns a value, Celery stores it there under the task ID. If someone
  later calls .get() on the task, Celery fetches from Redis.

  

    Ways to use the gateway

  1. Direct HTTP calls (curl / any language)

  2. Drop-in replacement for the OpenAI SDK (the headline feature)

  Since /v1/chat/completions and /v1/embeddings use OpenAI's request/response shape exactly, any OpenAI SDK just works
  if you point it at your gateway URL.

  Python:
  from openai import OpenAI

  client = OpenAI(
      api_key="sk-gw-live-YOUR_KEY",            # your gateway key, not OpenAI's
      base_url="https://<codespace>-80.app.github.dev/v1",
  )

  resp = client.chat.completions.create(
      model="gemini-2.5-flash",                  # or any model your gateway routes
      messages=[{"role": "user", "content": "Hello"}],
  )
  print(resp.choices[0].message.content)

  Zero code change between "talking to OpenAI" and "talking to your gateway." They get caching, semantic matching,
  Gemini fallback, rate limits, cost tracking — all transparently.

  What to watch out for on the free tier

  - Gemini API rate limits kick in fast. gemini-2.5-flash free tier is ~15 req/min. With 5 users hitting at full speed,
  you'll exhaust it within 30 seconds unless cache absorbs most requests. That's actually fine — it demonstrates exactly
   what your gateway saves you from.
  - Codespace CPU. 2 vCPUs handle ~30 RPS comfortably. Beyond that, latency climbs because of resource contention, not
  because of your gateway logic.
  - Rate limit (your own). Default is 60/minute per API key. With users=5 doing ~1 req/sec each, you'll be near the
  limit. Either use --users 3 or temporarily bump RATE_LIMIT_REQUESTS_PER_MINUTE in .env.prod.
